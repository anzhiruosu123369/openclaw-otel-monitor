# OpenClaw OTel Monitor — 产品迭代路线图

> 调研来源：LangFuse、LangSmith、Arize Phoenix、Grafana、Datadog

当前项目是一个**实时 Token 用量看板 + Gateway 健康监控**，但缺少 LLMOps 的核心层。以下是按产品价值排序的迭代路线。

---

## Phase 1 — 监控基建（落地 LLMOps 基础数据模型）

**当前状态：** 只追踪了 token 数，缺少 LLM 工程必需的数据模型

| # | 功能 | 价值 | 说明 |
|---|------|------|------|
| 1 | **分层 Trace 视图** | ⭐⭐⭐⭐⭐ | 当前 trace 是扁平列表，但 JSONL 事件已有 `parentId`。改为树形展开可跟踪因果链，一眼看出哪步是延迟瓶颈 |
| 2 | **美元费用追踪** | ⭐⭐⭐⭐⭐ | 内置模型定价表（OpenAI/Anthropic/Google/OpenClaw），自动把 token → USD。仪表盘加费用排行、费用趋势、Session 级别费用 |
| 3 | **生产级告警** | ⭐⭐⭐⭐⭐ | 可配置阈值（TPM 逼近限流、错误率飙升、费用超支）+ 通知渠道（Slack/Webhook/钉钉） |
| 4 | **打分与评测框架** | ⭐⭐⭐⭐ | SQLite 建 `scores` 表，支持 LLM-as-Judge、正则检查、人工标注。评测结果回到看板做质量趋势 |

## Phase 2 — 工程工作流（从"看"到"改"）

**当前状态：** 纯只读仪表盘，无法干预生产

| # | 功能 | 价值 | 说明 |
|---|------|------|------|
| 5 | **Prompt 管理与版本控制** | ⭐⭐⭐⭐⭐ | 数据库存 prompt 模板 + 版本。支持 `production`/`staging` 标签一键回滚。Link 到 trace 知道哪版 prompt 产生了什么输出 |
| 6 | **Playground —— 同屏调试** | ⭐⭐⭐⭐ | 选中 trace 中的某次 LLM 调用，修改 prompt/模型/参数，重跑并对比输出。不离开监控界面 |
| 7 | **数据集与回归实验** | ⭐⭐⭐⭐ | 从生产 trace 创建测试数据集。改 prompt 或模型后在数据集上批量跑实验，对比通过率/费用/延迟 |

## Phase 3 — 平台能力（协作与规模化）

**当前状态：** 单人单机

| # | 功能 | 价值 | 说明 |
|---|------|------|------|
| 8 | **Trace 对比视图** | ⭐⭐⭐⭐ | 两个 trace 并排 diff，高亮差异步骤。调试"为什么这次失败了" |
| 9 | **用户反馈收集** | ⭐⭐⭐ | 前端注入反馈按钮（👍/👎/自定义），反馈回写 trace 做评测信号 |
| 10 | **画布式自定义看板** | ⭐⭐⭐ | Grafana 式的拖拽布局，用户自行组合图表/统计卡片 |
| 11 | **数据导出与可编程 API** | ⭐⭐⭐ | CSV/JSON 导出；SQL 查询接口；Webhook 事件推送 |

---

## 建议启动顺序

```
Phase 1 (1-4)     →    Phase 2 (5-7)    →    Phase 3 (8-11)
  2-4 周                3-5 周                按需
```

**最优先（1-2 周可交付）：** #1 分层 Trace + #2 美元费用 + #4 评测框架，三个功能加起来把项目从"Token 计数器"升级为"LLM 工程平台"。
